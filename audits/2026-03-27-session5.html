B−  ·  Capable, Execution Gap

The simulation engine and persistence story are world-class for a browser game. Visual and UX quality has levelled up significantly across Sessions 3–4. But 4 core engines built in S4 are dormant — not called from the game loop — and the tutorial is fully built but never launched. Community and growth hooks remain at zero. The project is technically impressive; the gap is in activation of the work already done.

B−

Category Scores (10 × 10)

Simulation Engine Depth 8 / 10 →
QB depth ratings (short/medium/deep), route families, coverage shells, AV, PFR-calibrated realism, Monte Carlo regression, world-state gameplay effects. beatReporter, rivalryDNA, injurySystem, and gmLegacy engines all exist and are correct — but are not called from advanceWeek() or advanceSeason(), so they produce no live data. Deduct 2 for this integration gap.
Frontend / UX Quality 8 / 10 ↑
Command-deck treatment on Overview, Contracts, Scouting, Settings. Player dossier hero with generated portrait and contract context. Box score subtab UX, HoF/Season Awards galleries, team color theming, side-nav grouping. S4 added news ticker, cap panels, coaching DNA card, agent modal, shortcuts modal. Some legacy tabs not yet updated. app.js at 6,519 lines is a growing maintainability concern.
Mobile & Accessibility 6 / 10 →
375px CSS fix shipped. initMobileLoop() wired into init(). Keyboard shortcuts extended (W, N, 1–9, ?, Escape). No confirmed smoke test results for S4 panels at 375px — this is a pending human action. No formal accessibility audit. Screen-reader compliance unknown. Aria labels present in some places, not systematic.
Feature Integration (depth vs. breadth) 6 / 10 ↓
S4 shipped 15 features but 4 engine modules (beatReporter, rivalryDNA, injurySystem, gmLegacy) are not called from the game loop — their UI panels render empty or stale data. tutorialCampaign.js is fully built (252 lines, 3-step FTUE) but is never imported or called in app.js. Commissioner Lobby is async UI only with no real multiplayer backend. Dynasty Share outputs a print window with no persistent URL. These are not design failures — they are integration steps still pending.
Test Coverage & CI 7 / 10 →
23 unit test files covering most core systems (stats regression, realism, depth ratings, session actions, standings, calibration, determinism, pass structure, world-state, etc.). 2 Playwright UI specs. CI runs both on every push. The gap: new-systems.test.js tests S1/S2 systems. beatReporter, rivalryDNA, injurySystem, gmLegacy, and draftCombine have zero unit tests. S4 systems are untested.
Persistence & Data Integrity 9 / 10 ↑
Strongest category. Multiple adapter stack: localStorage (rolling backup), IndexedDB (250MB, S4), server-backed file saves, snapshot import/export. Rewind system (max 10 slots, auto-pruned) canonical in localApiRuntime.js. Snapshot migration versioning. Quota-limit recovery. Determinism test. Backup pruning before retry. Client + server save stores both tested. Deduct 1 for IndexedDB not yet exercised at Pages scale.
Onboarding & FTUE 5 / 10 ↑
tutorialCampaign.js is fully implemented — 3-step FTUE with choice-based archetype selection, progress pips, skip option, and localStorage gate. It is never imported or triggered. Setup hero has good framing text and feature ribbon cards. Season Preview Panel contextualises the first game week. No in-game guidance, no contextual tooltips, no "What's New" on first load after update.
Community & Growth Hooks 3 / 10 →
No analytics integration (no Plausible, Fathom, Umami, or equivalent). No Open Graph / social preview meta tags in index.html. No community links (Discord, Reddit, forum). No in-game feedback mechanism. Dynasty Share outputs a print window — no shareable URL, no social card, no image capture. Pre-launch is appropriate context, but these foundations should be in place before first external players arrive.
Deployment & Operations 7 / 10 →
CI (unit + UI tests), deploy-pages.yml, deploy-backend.yml all present. Node 24 action upgrades done. Setup timing diagnostics in place. Backend image tag normalization fixed. Pages direct-deploy is the canonical path. S3 + S4 work — 20 features across 2 sessions — is still local only and has not been deployed. The longer this sits, the higher the merge/conflict risk. Deduct 2 for sitting undeployed.
Project Health & Process 8 / 10 →
LATEST_HANDOFF.md is thorough. TASK_BOARD clearly maintains Now/Next/Later + SIL items. SIL scoring active (sparkline ▁▆▇, clear upward trend). Studio OS compliance high. CURRENT_STATE.md is significantly stale — it still describes the state from before Session 3 and does not mention any of the 20 features shipped in S3+S4. DECISIONS.md is sparse. Context overhead growing; may benefit from pruning older log entries.

SWOT Analysis

Strengths
  • Simulation depth genuinely exceeds Football-GM / ZenGM — world-state, QB depth ratings, route families, AV
  • Persistence stack is robust (4 adapter paths, rewind, versioned migration)
  • 23 unit test files + 2 Playwright specs + full CI
  • Command-deck UI surfaces are distinctive and polished for a browser game
  • PFR-calibrated realism baseline is a hard moat — months of data work behind it
  • beatReporter, rivalryDNA, gmLegacy, injurySystem are correctly designed — just need wiring
Weaknesses
  • 4 core S4 engines dormant — no live data flows through news ticker, rivalry, or GM legacy card
  • Tutorial campaign built but never launched — biggest missed onboarding win
  • Community / growth = zero: no analytics, no OG tags, no community presence
  • CURRENT_STATE.md is 2+ sessions stale — context debt
  • app.js monolith (6,519 lines) and growing — split complexity risk
  • 20 features undeployed across 2 sessions — compounding deployment risk
Opportunities
  • Activating 4 engine hooks is a single-session task that unlocks narrative richness, injury realism, and legacy progression simultaneously
  • Tutorial activation converts the setup hero experience into a proper FTUE — one import + one function call
  • Analytics baseline (Plausible/Umami) before first external users = ability to learn from beta
  • Dynasty Share → real shareable URL/image card is a high word-of-mouth lever
  • Injury + Beat Reporter live data = the richest gamefeel upgrade available without new design
  • Commissioner Lobby → async league play differentiates from every browser competitor
Threats / Risks
  • Beta launch (2026-06-01) without community hooks = no retention signal and no feedback loop
  • Engine hooks staying dormant another session = risk of narrative systems atrophying (stale data visible in UI)
  • app.js monolith growth could make future UI work increasingly expensive
  • IndexedDB adapter untested at Pages scale — quota/browser compat surprises possible at launch
  • No changelog or "what's new" surface means returning beta users won't know what changed
  • Undeployed S3+S4 work means beta testers see an outdated product

Key Recommendations (Short)

🔴 Deploy immediately — S3+S4 work is 2 sessions behind
Push to main, confirm Pages CI green, verify live URL. Every day this sits local is compounding deployment risk and means beta testers see an outdated product. This is a human action — cannot be automated.
🔴 Wire 4 engine hooks into GameSession in this session
beatReporter.reportWeeklyResults() → advanceWeek() after game sim.
rivalryDNA.recordWeekRivalries() → advanceWeek() after game results.
injurySystem.rollInjuryChecks() → advanceWeek() after simulation.
gmLegacyScore.updateGmLegacyAfterSeason() → postseason phase.
Until these run, 4 UI panels show empty or stale data on every advance. Estimated effort: 1–2 hours.
🔴 Wire tutorialCampaign.js into app.js — one import, one call
Add import { startTutorial } from "./lib/tutorialCampaign.js" and call it after first league load if the tutorial hasn't been seen. This is 2–3 lines of code and unlocks the full 3-step FTUE that was already built and tested.
🟡 Add OG meta tags + a privacy-friendly analytics script before beta
5 meta tags in index.html (og:title, og:description, og:image, og:url, twitter:card) unlock social preview when dynasty timelines are shared. One Plausible/Umami script tag gives you a beta feedback signal from day one. Combined effort: under 1 hour.
🟡 Refresh CURRENT_STATE.md — it's 2 sessions stale
The file still reflects pre-Session 3 state. It's the "live snapshot" field that future sessions rely on. Updating it now prevents compounding drift. Can be done at closeout.
🟢 Add unit tests for S4 engine systems
beatReporter, rivalryDNA, injurySystem, gmLegacy, and draftCombine have zero test coverage. A basic smoke test for each (init, one result cycle, output shape check) would add confidence before they power live gameplay.
🟢 Plan an app.js split before it becomes a blocker
At 6,519 lines the file is a growing risk. It doesn't need splitting today, but scoping a panel-extraction plan (e.g. cap-panels.js, narrative-panels.js) before the next feature block will save compounding pain.
